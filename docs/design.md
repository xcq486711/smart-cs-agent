# 智能客服系统 — 设计文档

**版本**: v0.1
**创建日期**: 2026-06-03
**基于**: Hello-Agents 第 1 章 — 初识智能体

## 1. 项目愿景

构建一个"禁得起敲打"的多领域智能客服 Agent，支持医疗咨询、旅游规划等场景。
不是 demo，是真正能处理复杂对话、调用外部工具、动态推理的 AI Native Agent。

## 2. PEAS 模型分析

### 2.1 医疗咨询领域

| PEAS 维度 | 描述 |
|---|---|
| **P (性能度量)** | 分诊准确率 > 90%、平均响应时间 < 5s、关键症状不遗漏率 100%、用户满意度 > 4.5/5 |
| **E (环境)** | 对话上下文、医学知识库（RAG）、症状-科室映射表、挂号系统接口 |
| **A (执行器)** | 生成回复文本、查询知识库、调用挂号 API、转接人工、标记紧急情况 |
| **S (传感器)** | 用户症状描述（自然语言）、既往病史、检查报告（未来支持图片） |

### 2.2 旅游规划领域

| PEAS 维度 | 描述 |
|---|---|
| **P (性能度量)** | 行程满意度、推荐相关性评分、预订完成率 |
| **E (环境)** | 天气 API、航班/酒店预订系统、景点信息库、地图服务 |
| **A (执行器)** | 生成行程方案、查询实时天气/票价、调用预订接口 |
| **S (传感器)** | 用户偏好描述、预算范围、出行日期、人数 |

## 3. 环境特性分析（为什么 Agent 模式更合适）

| 环境特性 | 医疗咨询 | 旅游规划 | 为什么 Workflow 搞不定 |
|---|---|---|---|
| **部分可观察** | 看不到用户真实病情全貌 | 看不到所有可选航班 | 需要多次追问 + 逐步收集信息 |
| **随机性** | 号源实时变化 | 价格、库存动态变化 | 需要处理不确定性 + 提供备选 |
| **多 Agent** | 其他用户也在抢号 | 其他用户也在订票 | 需要快速响应 |
| **序贯且动态** | 当前症状询问影响后续判断 | 当前推荐影响后续选择 | 每一步依赖上一步 |

## 4. 架构思路：Workflow + Agent 混合

```
┌─────────────────────────────────────────────┐
│                 智能客服系统                    │
│                                               │
│  ┌──────────┐    ┌──────────┐    ┌─────────┐ │
│  │ Workflow  │    │  Agent   │    │ 规则兜底 │ │
│  │ 层         │    │  核心     │    │ 层       │ │
│  │           │    │          │    │          │ │
│  │ "退货政策" │───→│ 动态推理  │───→│ 退款确认  │ │
│  │  FAQ 快速  │    │  工具调用  │    │  人工审核  │ │
│  │  响应      │    │  多步规划  │    │  合规检查  │ │
│  └──────────┘    └──────────┘    └─────────┘ │
│                                               │
│  简单问题 → Workflow 直通回答                    │
│  复杂问题 → Agent 动态推理 → 关键操作规则把关       │
└─────────────────────────────────────────────┘
```

**设计原则**: LLM 当大脑（模糊推理），Workflow 当骨架（关键路径），规则当底线（安全兜底）。

## 5. 第 2 章启示：智能的"进化阶梯"如何指导架构选型

### 三层能力模型

学习 Agent 发展史后，将智能客服架构重新建模为三层：

```
┌──────────────────────────────────────┐
│  Layer 3: LLM Agent（联结主义+预训练）│
│  - 复杂多步推理                       │
│  - 模糊意图理解                       │
│  - 动态规划+工具调用                   │
│  例："我头疼三天了，怎么办？"           │
├──────────────────────────────────────┤
│  Layer 2: 知识检索 + 推理（符号主义+RL）│
│  - 结构化知识库查询                    │
│  - 规则链推理（症状→科室→挂号）        │
│  - 置信度评分                         │
│  例：识别"太阳穴疼+无发烧"→偏头痛可能  │
├──────────────────────────────────────┤
│  Layer 1: 规则引擎（符号主义）          │
│  - FAQ 快速匹配                       │
│  - 合规检查（金额/权限/敏感操作）       │
│  - 关键词路由                         │
│  例："退货政策是什么？"→直接返回标准答案 │
└──────────────────────────────────────┘
```

### 选型决策树（每次收到用户消息时走一遍）

```
用户消息
    │
    ▼
能匹配 FAQ 规则？ ──是──→ Layer 1 直接回复（速度优先，零 LLM 成本）
    │ 否
    ▼
需要查结构化知识库？ ──是──→ Layer 2: RAG 检索 + 规则推理
    │ 否
    ▼
需要多步推理/工具调用？ ──是──→ Layer 3: LLM Agent
    │
    ▼
最终输出前 → Layer 1 兜底检查（涉敏/涉险/涉法？→人工确认）
```

### 关键启示

- **ELIZA 的教训**：纯规则系统永远无法覆盖开放域对话。FAQ 层只处理"已知问题"，边界外必须交给 LLM
- **MYCIN 的教训**：即使规则推理准确率高，如果知识库维护成本无穷大，系统也无法规模化
- **SHRDLU 的教训**：封闭世界里的智能≠开放世界里的智能。Agent 必须能处理"我没想到你会问这个"
- **心智社会的教训**：不要把希望寄托在"一个完美的 AI"上。多 Agent 协作 + 各司其职 > 单一巨型 Agent
